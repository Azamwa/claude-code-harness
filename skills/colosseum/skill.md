# 콜로세움 모드 ON

이 시점부터 **콜로세움 모드**가 활성화된다. 사용자가 작업을 요청하면 아래 프로세스를 따른다.
단순 질문, 설명 요청 등 비작업성 대화에는 적용하지 않는다.

## 전제

- 메인 에이전트 1개 (Opus): 전체 세션 유지, 기획/코딩/수정 담당
- 기획 단계와 리뷰 단계에서 Agent Teams를 활용하여 팀메이트 간 양방향 토론을 수행
- 테스트 실행은 별도 Sub-agent(Sonnet)로 처리 (토론 불필요)
- 팀메이트는 각 단계 완료 후 종료

## 언어 규칙

- **모든 팀메이트는 반드시 한글로 출력한다.**
- 팀메이트 생성 시 프롬프트에 다음 문구를 반드시 포함한다:
  > "모든 분석, 리뷰, 토론, 메시지를 반드시 한글로 작성하라. 코드 블록과 기술 용어(변수명, 함수명, 라이브러리명 등)만 영어를 허용한다."
- 메인 에이전트의 종합 리포트도 한글로 작성한다.

## 워크플로우

### [1] 기획 (메인, Opus, Plan Mode)

#### 1-1. 요구사항 정리
- 사용자 + AI 협업
- 페이지 단위로 요구사항 정리

#### 1-2. 설계 방향 제안
- 메인이 구현 방안 2~3개를 비교하여 제시
- 각 방안의 리스크, 대안, 누락 사항을 함께 제시 (self-critique)
- 제시 후 사용자에게 다음 선택지를 제공한다:

```
설계 방안을 제시했습니다. 다음 중 선택해주세요:

1. 팀 토론 — architect 팀을 소집하여 방안별 찬반 토론 후 결정
2. 바로 합의 — 이 중 하나를 선택하고 코딩으로 진행
```

#### 1-3a. 팀 토론 (사용자가 1번 선택 시)

`TeamCreate`로 설계 팀을 생성한다. 팀 이름: `design-{작업명}`

Agent tool로 팀메이트 2명을 **동시에** 생성한다:

| 팀메이트 name | 모델 | subagent_type | 역할 |
|---------------|------|---------------|------|
| `architect-a` | Opus | general-purpose | 방안 A를 옹호하며 설계 근거 제시 |
| `architect-b` | Opus | general-purpose | 방안 B를 옹호하며 설계 근거 제시 |

팀메이트 생성 시 프롬프트에 반드시 포함:
- 요구사항 전문
- 자신이 옹호할 방안과 그 이유
- 상대 방안의 정보 (공정한 비교를 위해)
- 프로젝트의 기술 스택과 기존 패턴 정보

팀메이트는 **서로 직접 SendMessage**로 토론한다. 메인은 중계하지 않는다.

팀메이트 프롬프트에 다음 지시를 포함한다:
> "독립 분석이 끝나면 상대 팀메이트에게 직접 SendMessage로 자신의 분석을 보내라.
> 상대의 메시지를 받으면 반론/보강하여 다시 직접 보내라.
> 총 3라운드 교환 후, 최종 입장을 메인(team-lead)에게 보내라."

교차 토론 (3라운드, 팀메이트 간 직접 대화):
- **Round 1**: 각자 분석 완료 후 상대에게 직접 전송 → 상대 방안의 리스크/약점 지적 + 자기 방안의 우위 근거
- **Round 2**: 상대 반론을 받고 직접 재반박 또는 양보
- **Round 3**: 최종 입장 정리 → 메인(team-lead)에게 결과 전달

3라운드 종료 후 메인이 토론 결과를 종합하여 사용자에게 제시:

```
## 설계 토론 결과

| 항목 | 방안 A | 방안 B |
|------|--------|--------|
| 핵심 주장 | ... | ... |
| 인정한 약점 | ... | ... |
| 합의된 점 | ... |
| 미합의 쟁점 | ... |

추천: [근거와 함께 메인의 추천안]
```

사용자가 최종 방향을 선택한다.
팀을 shutdown → `TeamDelete`로 정리한다.

#### 1-3b. 바로 합의 (사용자가 2번 선택 시)

사용자가 방안을 선택하고 바로 진행한다.

#### 1-4. 작업 분할
- 합의된 방향을 기반으로 작업을 A, B, C, D 단위로 분할
- 기획 결과는 `.claude/plans/` 디렉토리에 파일로 저장 (컨텍스트 압축 대비)

### 이하 작업 단위(A→B→C→D) 루프

### [2] 코딩 (메인, Opus)

- 합의된 기획 기반으로 한 작업 단위씩 진행
- **2-1. 구현**: 기능 코드 작성
- **2-2. 테스트 작성**: 구현 완료 후 테스트 코드 작성 (관점 전환: "뭐가 깨질 수 있을까")

### [3] 팀 리뷰 (Agent Teams + Sub-agent)

#### 3-1. 팀 생성

`TeamCreate`로 리뷰 팀을 생성한다. 팀 이름: `review-{작업단위}` (예: `review-a`)

그 후 Agent tool로 팀메이트 2명을 **동시에** 생성한다:

| 팀메이트 name | 모델 | subagent_type | 역할 |
|---------------|------|---------------|------|
| `code-reviewer` | Opus | general-purpose | 구현 + 테스트 코드의 구조, 설계 패턴, 컨벤션 검토 |
| `security-reviewer` | Opus | general-purpose | 보안 취약점, 성능 이슈, 엣지케이스 검토 |

별도로 테스트 러너는 **일반 Sub-agent(Sonnet, 팀 외부)** 로 병렬 생성한다. 토론이 필요 없으므로 팀에 포함하지 않는다.

팀메이트 생성 시 각 에이전트에게 전달할 프롬프트에 반드시 포함:
- 리뷰 대상 파일 목록과 경로
- 구현 의도 및 기획 요약
- 리뷰 관점 (각자의 역할에 맞는 체크리스트)
- **교차 검증 라운드가 있으므로 자신의 리뷰를 명확하게 정리해둘 것**

#### 3-2. 독립 리뷰 → 교차 검증 (3라운드, 팀메이트 간 직접 대화)

팀메이트는 **서로 직접 SendMessage**로 토론한다. 메인은 중계하지 않는다.

팀메이트 프롬프트에 다음 지시를 포함한다:
> "독립 리뷰가 끝나면 상대 팀메이트에게 직접 SendMessage로 자신의 리뷰를 보내라.
> 상대의 메시지를 받으면 동의/반박/추가 지적하여 다시 직접 보내라.
> 총 3라운드 교환 후, 최종 리뷰 요약을 메인(team-lead)에게 보내라."

각 라운드에서 리뷰어는:
- 상대의 지적에 **동의**하면 → 근거를 보강하거나 추가 관점 제시
- 상대의 지적에 **반대**하면 → 구체적 코드 근거와 함께 반박 + 대안 제시
- 상대가 **놓친 이슈**가 있으면 → 새로 지적

라운드 진행:
- **Round 1**: 각자 독립 리뷰 완료 후 상대에게 직접 전송
- **Round 2**: 상대 리뷰를 받고 직접 반론/보강/합의
- **Round 3**: 최종 입장 정리. 합의되지 않은 항목은 "미합의"로 표시 → 메인(team-lead)에게 결과 전달

3라운드 종료 후, 각 리뷰어가 메인에게 보내는 **최종 리뷰 요약**:
- 합의된 이슈 목록 (심각도 표시: Critical / Major / Minor)
- 미합의 이슈 목록 (각자의 입장 병기)

#### 3-4. 종합

메인이 다음을 종합하여 사용자에게 리포트한다:

```
## 리뷰 결과

### 합의된 이슈
| # | 심각도 | 내용 | 수정 방향 |

### 미합의 이슈
| # | code-reviewer 의견 | security-reviewer 의견 |

### 테스트 결과
- 통과: X개 / 실패: Y개
- 실패 항목: ...
```

미합의 이슈는 사용자가 판단한다.
리포트 후 팀을 `SendMessage`로 shutdown → `TeamDelete`로 정리하고 [4]로 넘어간다.

### [4] 수정 (메인, Opus)

- 합의된 리뷰 피드백 + 사용자가 결정한 미합의 항목 + 테스트 실패 반영
- 메인이 직접 수정 (코딩 컨텍스트 유지)

### [5] 최종 확인 (메인)

- tsc --noEmit (타입 검사)
- eslint — 변경된 파일 대상 (`git diff --name-only` 기반, ts/tsx만)
- 테스트 재실행
- 3개 모두 통과해야 다음 작업 단위로 진행 ([2]로 복귀)

## 컨텍스트 관리

- 대화 중반: `/summarize`로 진행 상황 확인
- 대화 후반 (느려질 때): `/compact`로 컨텍스트 압축 후 계속 진행
- 기획 결과는 파일로 저장해두므로 압축/새 대화 시에도 복원 가능

## 모드 해제

다음 조건에서 콜로세움 모드가 해제된다:

- 사용자가 **"일반모드"** 라고 말하면 즉시 해제
- 새 대화에서는 기본 비활성 (매 대화마다 `/colosseum`으로 명시적 활성화 필요)

해제 시 안내:

```
콜로세움 모드 OFF — 일반 모드로 전환합니다.
```

## 모드 활성화 안내

사용자에게 다음과 같이 안내한다:

```
콜로세움 모드 ON — 작업을 요청하면 [1] 기획부터 시작합니다. 해제: "일반모드"
```
