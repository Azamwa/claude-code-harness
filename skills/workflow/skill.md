# 워크플로우 모드 ON

이 시점부터 **워크플로우 모드**가 활성화된다. 사용자가 작업을 요청하면 아래 프로세스를 따른다.
단순 질문, 설명 요청 등 비작업성 대화에는 적용하지 않는다.

## 전제

- 메인 에이전트 1개 (Opus): 전체 세션 유지, 기획/코딩/수정 담당
- Sub-agent는 병렬 리뷰 단계에서만 생성 → 결과 리턴 → 소멸
- Sub-agent는 컨텍스트 격리됨 (부모 대화 상속 X)
- 에이전트 간 토의 X, 일방향 검증만 사용

## 워크플로우

### [1] 기획 (메인, Opus, Plan Mode)

- 사용자 + AI 협업
- 페이지 단위로 요구사항 정리
- 구현 방안 2~3개 비교 후 방향 합의
- 각 방안의 리스크, 대안, 누락 사항을 함께 제시 (self-critique)
- 작업을 A, B, C, D 단위로 분할
- 기획 결과는 `.claude/plans/` 디렉토리에 파일로 저장 (컨텍스트 압축 대비)

### 이하 작업 단위(A→B→C→D) 루프

### [2] 코딩 (메인, Opus)

- 합의된 기획 기반으로 한 작업 단위씩 진행
- **2-1. 구현**: 기능 코드 작성
- **2-2. 테스트 작성**: 구현 완료 후 테스트 코드 작성 (관점 전환: "뭐가 깨질 수 있을까")

### [3] 병렬 리뷰 (Sub-agent 3개 동시 생성)

| Agent | 모델 | 권한 | 역할 |
|-------|------|------|------|
| 코드 리뷰 | Opus | 읽기 전용 | 구현 + 테스트 코드의 구조, 설계 패턴, 컨벤션 검토 |
| 보안/성능 리뷰 | Opus | 읽기 전용 | 구현 + 테스트 코드의 보안 취약점, 성능 이슈, 엣지케이스 |
| 테스트 실행 | Haiku | 읽기 + Bash | 테스트 실행 및 결과 리포트 (작성 X, 실행만) |

- 3개를 **반드시 단일 메시지에서 동시에(병렬로)** 생성
- 3개 결과를 메인에게 종합 리포트

### [4] 수정 (메인, Opus)

- 리뷰 피드백 + 테스트 실패 반영
- 메인이 직접 수정 (코딩 컨텍스트 유지)

### [5] 최종 확인 (메인)

- tsc --noEmit (타입 검사)
- eslint — 변경된 파일 대상 (`git diff --name-only` 기반, ts/tsx만)
- 테스트 재실행
- 3개 모두 통과해야 다음 작업 단위로 진행 ([2]로 복귀)

## 컨텍스트 관리

- 대화 중반: `/summarize`로 진행 상황 확인
- 대화 후반 (느려질 때): `/compact`로 컨텍스트 압축 후 계속 진행
- 기획 결과는 파일로 저장해두므로 압축/새 대화 시에도 복원 가능

## 모드 해제

다음 조건에서 워크플로우 모드가 해제된다:

- 사용자가 **"일반모드"** 라고 말하면 즉시 해제
- 새 대화에서는 기본 비활성 (매 대화마다 `/workflow`로 명시적 활성화 필요)

해제 시 안내:

```
워크플로우 모드 OFF — 일반 모드로 전환합니다.
```

## 모드 활성화 안내

사용자에게 다음과 같이 안내한다:

```
워크플로우 모드 ON — 작업을 요청하면 [1] 기획부터 시작합니다. 해제: "일반모드"
```
