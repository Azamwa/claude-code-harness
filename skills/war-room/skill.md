# War Room 모드

버그/에러가 반복 수정에도 해결되지 않을 때, 다수의 에이전트가 협업하여 문제를 해결하는 프로세스.

## 전제

- 사용자가 `/war-room`으로 수동 발동
- 워크플로우 모드와 독립적으로 사용 가능 (워크플로우 활성화 여부 무관)
- 모든 에이전트는 Opus 사용
- 사용자가 증상, 시도한 수정, 실패 이유 등 맥락을 제공해야 함

## 플로우

### [W-1] 분석 회의 (Sub-agent 3개, Opus, 병렬)

메인이 다음 정보를 정리하여 3개 에이전트에게 **동일한 컨텍스트**를 전달한다:

- 현재 증상 (무엇이 잘못되고 있는지)
- 관련 코드 파일 목록
- 지금까지 시도한 수정과 그 결과
- 사용자가 제공한 추가 맥락

각 에이전트는 **독립적으로** 다음을 수행한다:

| Agent | 관점 | 역할 |
|-------|------|------|
| 분석가 A | 직접 원인 | 코드 흐름을 추적하여 버그의 직접적 원인 특정 + 최소 수정안 제시 |
| 분석가 B | 구조적 원인 | 아키텍처/설계 레벨에서 근본 원인 분석 + 구조 개선안 제시 |
| 분석가 C | 사이드이펙트 | A, B가 제안할 수 있는 수정의 부작용 예측 + 대안 제시 |

3개를 **반드시 단일 메시지에서 동시에(병렬로)** 생성한다.

### [W-2] 결정 (메인 + 사용자)

메인이 3개 결과를 종합하여 사용자에게 비교표로 제시한다:

```
| 제안 | 원인 분석 | 수정 방법 | 사이드이펙트 | 난이도 |
```

사용자와 방향을 합의한 후 진행한다.

### [W-3] 수정 (Sub-agent 1개, Opus)

합의된 해결책을 **새로운 에이전트**가 구현한다.

이 에이전트에게 전달하는 프롬프트에 반드시 포함할 내용:
- W-1에서 합의된 원인 분석 요약
- 합의된 수정 방향
- W-1에서 예측한 사이드이펙트 (주의사항으로)
- 수정 대상 파일 목록

### [W-4] 검증 리뷰 (Sub-agent 3개, Opus, 병렬)

W-1에서 분석했던 **동일한 3가지 관점**으로 수정 결과를 검증한다:

| Agent | 검증 내용 |
|-------|----------|
| 검증자 A | 직접 원인이 실제로 해결되었는지 코드 레벨 확인 |
| 검증자 B | 구조적 문제가 해결되었는지, 새로운 설계 이슈가 없는지 확인 |
| 검증자 C | W-1에서 예측한 사이드이펙트가 실제로 방지되었는지 확인 |

3개를 **반드시 단일 메시지에서 동시에(병렬로)** 생성한다.

### [W-5] 최종 확인 (메인)

- tsc --noEmit (타입 검사)
- eslint (린트 검사)
- 관련 테스트 실행
- 모두 통과 시 War Room 종료

## 종료 조건

- W-5 통과 시 자동 종료
- 사용자가 "종료" 또는 "해산"이라고 하면 즉시 종료

종료 시 안내:

```
War Room 종료 — 문제가 해결되었습니다.
```

## 모드 활성화 안내

사용자에게 다음과 같이 안내한다:

```
War Room 소집 — 증상과 맥락을 설명해주세요.
```
